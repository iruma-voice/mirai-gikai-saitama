## 前提・制約

- 現時点では**党公式ではなく自費運営**のため、コストを最小限に抑えたい
- 月額の**予算上限を設定でき、上限を超えたら止まる**（予算超過が発生しない）運用が必須
- 従量課金型で、**一定量は無料**であることが望ましい
- 本番環境への段階的移行を想定（まず動作確認できる状態で公開 → 必要に応じてスケールアップ）

## 構成サービスとその選択肢

このアプリは以下のサービスで構成される：

| 役割 | ローカル | 本番候補 |
|------|---------|---------|
| Web フロントエンド (Next.js) | localhost:3000 | **Vercel** |

| AI プロンプト管理 | Langfuse Cloud | **Langfuse Cloud**（既存） |

## 推奨構成: Vercel + Supabase Cloud

### なぜ Vercel か

元の「みらい議会」も Vercel を利用しており、このプロジェクトも `.vercel` 設定や `@vercel/speed-insights` など Vercel 前提の実装が含まれている。
Next.js との親和性が高く、設定が最小限で済む。

### コスト試算（月額）

#### Vercel

| プラン | 月額 | 内容 |
|--------|------|------|
| Hobby（無料） | $0 | 個人利用。商用不可。帯域 100GB/月。Serverless Function 実行時間 100GB-hours |
| Pro | $20/人（最低$20） | 商用利用可。帯域 1TB/月。Serverless Function 1,000GB-hours |

**推奨: Pro プラン（$20/月 ≒ 約3,000円/月）**

理由：
- Hobby は商用利用が利用規約上禁止（政党関連サービスは商用扱いになりうる）
- 月固定 $20 で上限が明確
- 使いすぎによる追加課金は帯域超過時のみ（$0.15/GB）

> **Vercel の予算制限について:**
> Vercel Pro では「Spend Management」機能で**月の支出上限を設定**できる。
> 上限に達すると追加のデプロイや Serverless 実行が停止するため、青天井にはならない。
> 設定箇所: Vercel Dashboard → Settings → Billing → Spend Management

#### Supabase Cloud

| プラン | 月額 | 内容 |
|--------|------|------|
| Free | $0 | DB 500MB、API 50万リクエスト/月、ストレージ 1GB |
| Pro | $25/月 | DB 8GB、API 無制限、ストレージ 100GB |

**推奨: まず Free プランで運用開始**

理由：
- 川崎市議会の議案数・アクセス数は当初は少ないため無料枠で十分
- Free プランは**1週間アクティビティがないとプロジェクトが一時停止**される仕様あり
  → 定期的なアクセスがある場合は問題なし。心配なら Pro ($25/月) へ
- Free プランは組織1つにつき2プロジェクトまで

> **Supabase の使用量制限:**
> Free プランは使用量上限に達すると自動的に新規リクエストを制限する（DB書き込み停止等）。
> 追加課金は発生しない。

#### AI API（チャット機能）

| 方式 | 内容 | コスト |
|------|------|--------|
| Vercel AI Gateway 経由 | Anthropic API をゲートウェイ越しに呼ぶ | Anthropic API 従量課金 + Vercel Gateway 費用 |
| Anthropic API 直接 | Claude API を直接呼ぶ | Anthropic API 従量課金のみ |

**AIのコスト管理:**

アプリに `CHAT_DAILY_COST_LIMIT_USD` という**日次コスト上限機能**がすでに実装されている。
本番では適切な値（例: $1〜$5/日）を設定することで月の AI コストを管理できる。

Anthropic API の概算コスト：

| 想定アクセス | チャット利用回数/日 | 概算月額 |
|------------|-----------------|--------|
| 少量（試験運用） | 10〜50回 | $1〜$5 |
| 中量 | 100〜500回 | $10〜$50 |

## 展開手順（詳細）

### 1. Supabase Cloud プロジェクト作成

1. https://supabase.com でアカウント作成・組織作成
2. 新規プロジェクト作成（無料プランで開始）
3. ローカルのマイグレーションを適用: `npx supabase db push`
4. 本番の接続情報を控える（後述の「必要な環境変数」を参照）

> **Supabase の API キーについて:**
> Supabase Dashboard の Settings → API に以下が表示される：
> - **Project URL** → `NEXT_PUBLIC_SUPABASE_URL` および `SUPABASE_URL` に使用
> - **anon / public key** → `NEXT_PUBLIC_SUPABASE_ANON_KEY` に使用
> - **service_role key** → `SUPABASE_SERVICE_ROLE_KEY` に使用（管理者操作用）

### 2. Vercel プロジェクト作成（Web）

1. https://vercel.com でアカウント作成・Pro プランに課金
2. 「Add New Project」→ GitHub リポジトリを接続
3. **以下の設定を必ず行う（デフォルトのままでは動かない）:**

   | 設定項目 | 値 |
   |---------|-----|
   | Framework Preset | **Next.js**（自動検出されない場合は手動で選択） |
   | Root Directory | `web` |
   | Install Command | `cd .. && pnpm install --frozen-lockfile` |
   | Build Command | `pnpm run build`（デフォルト） |

4. 環境変数を設定（後述の「必要な環境変数」を参照）
5. 自動デプロイは無効化済み（`vercel.json` の `"deploymentEnabled": false`）。Deploy Hook 経由のみでデプロイされる

### 3. Vercel プロジェクト作成（Admin）

1. 別プロジェクトとして作成
2. **以下の設定を必ず行う:**

   | 設定項目 | 値 |
   |---------|-----|
   | Framework Preset | **Next.js**（手動で選択） |
   | Root Directory | `admin` |
   | Install Command | `cd .. && pnpm install --frozen-lockfile` |
   | Build Command | `pnpm run build`（デフォルト） |

3. 環境変数を設定
4. **Admin のアクセス制限:** Supabase 認証（メール・パスワード）+ `app_metadata.roles: ["admin"]` で保護
5. **Admin を一時的に外部から閉じたい場合:** Vercel Dashboard → 対象プロジェクト → Settings → Deployment Protection → Password Protection を ON にする（Pro プランの機能）

### 4. Admin ユーザーの作成

シードスクリプト（`pnpm seed`）は DB テーブルにデータを投入するが、**Supabase Auth ユーザーは作成しない**。
Admin ページにログインするには、Supabase Dashboard で手動作成が必要。

**手順:**

1. Supabase Dashboard → **Authentication** → **Users** → **「Add user」** → **「Create new user」**
   - Email: 任意のメールアドレス
   - Password: 強力なパスワードを設定

2. Supabase Dashboard → **SQL Editor** で以下を実行し、admin ロールを付与：

   ```sql
   UPDATE auth.users
   SET raw_app_meta_data = jsonb_build_object(
     'provider', 'email',
     'providers', ARRAY['email'],
     'roles', ARRAY['admin']
   )
   WHERE email = 'your-admin@example.com';
   ```

3. 確認：

   ```sql
   SELECT email, raw_app_meta_data FROM auth.users WHERE email = 'your-admin@example.com';
   ```

   `raw_app_meta_data` に `"roles": ["admin"]` が含まれていれば OK。

### 5. Spend Management の設定（Vercel）

Vercel Dashboard → Settings → Billing → Spend Management で月の支出上限を設定。

### 6. シードデータの投入

```bash
# 本番 DB に接続してシードを投入
SUPABASE_URL=<本番URL> SUPABASE_SERVICE_ROLE_KEY=<本番キー> pnpm seed
```

> シードは議案・会期・会派などのサンプルデータのみ投入する。Auth ユーザーは作成されない（手順4で手動作成）。

## 必要な環境変数

### Web（Vercel web プロジェクト）

| 変数名 | 値の取得先 | 説明 |
|--------|-----------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase → Settings → API → Project URL | 公開用 Supabase URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase → Settings → API → anon key | 公開用 API キー |
| `SUPABASE_URL` | Supabase → Settings → API → Project URL | サーバー側 admin クライアント用（`NEXT_PUBLIC_` なし・同じ値でよい） |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase → Settings → API → service_role key | サーバー側管理操作用（非公開） |
| `REVALIDATE_SECRET` | 自分で生成（後述） | Admin からのキャッシュ無効化リクエストの認証用シークレット |
| `LANGFUSE_SECRET_KEY` | Langfuse Dashboard | プロンプト管理・ログ |
| `LANGFUSE_PUBLIC_KEY` | Langfuse Dashboard | プロンプト管理・ログ |
| `LANGFUSE_BASEURL` | Langfuse Dashboard | Cloud の場合は `https://cloud.langfuse.com` |
| `CHAT_DAILY_COST_LIMIT_USD` | 任意の値（例: `5`） | AI チャットの日次コスト上限（USD） |
| `OPENAI_API_KEY` | OpenAI / Vercel AI Gateway | AI チャット用 API キー |

### Admin（Vercel admin プロジェクト）

| 変数名 | 値 | 説明 |
|--------|-----|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Web と同じ | |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Web と同じ | |
| `SUPABASE_URL` | Web と同じ | |
| `SUPABASE_SERVICE_ROLE_KEY` | Web と同じ | |
| `NEXT_PUBLIC_WEB_URL` | 本番 Web の URL（例: `https://xxx.vercel.app`） | Admin がキャッシュ無効化リクエストを送る先 |
| `REVALIDATE_SECRET` | Web と同じ値 | キャッシュ無効化リクエストの認証用（Web と一致させること） |

> **`SUPABASE_URL` について:**
> `packages/supabase/src/admin.ts` の `createAdminClient()` は `process.env.SUPABASE_URL`（`NEXT_PUBLIC_` なし）を参照する。
> `NEXT_PUBLIC_SUPABASE_URL` とは別に設定が必要。同じ URL 値を設定してよい。

> **`REVALIDATE_SECRET` について:**
> `.env.example` の `your-secret-key-here` はプレースホルダー。**必ず変更すること**。
> ランダムな文字列を生成して設定する：
> ```bash
> openssl rand -hex 32
> ```
> Web・Admin 両方に**同じ値**を設定する。

## 運用フロー：本番 DB へのデータ反映

AI情報収集などで収集したデータを本番 DB に反映する手順。

### 1. `.env.production` を作成（初回のみ）

```bash
cp .env.production.example .env.production
# エディタで本番の値を記入
```

`.env.production` に設定する内容：

```env
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_SERVICE_ROLE_KEY=<service_role_key>
NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon_key>
NEXT_PUBLIC_WEB_URL=https://<web-の-vercel-url>
REVALIDATE_SECRET=<REVALIDATE_SECRETと同じ値>
CHAT_DAILY_COST_LIMIT_USD=5
NEXT_PUBLIC_WEB_URL=http://localhost:3000
```

### 2. 本番 DB に向けてローカルの Admin を起動

```bash
pnpm dev:admin:prod-db
```

`localhost:3001` で Admin が起動し、操作内容が**本番 DB に直接書き込まれる**。

### 3. Admin 画面で操作 → 自動でキャッシュクリア

Admin 画面からの更新・削除操作は `invalidateWebCache()` を自動で呼び出すため、
Web 側のキャッシュが即時クリアされる（`NEXT_PUBLIC_WEB_URL` と `REVALIDATE_SECRET` が正しく設定されている場合）。

### キャッシュの手動クリア（Admin を経由しない操作後）

Supabase Dashboard や SQL で直接 DB を操作した場合は手動でキャッシュをクリアする：

```bash
curl -X POST https://<web-の-vercel-url>/api/revalidate \
  -H "Authorization: Bearer <REVALIDATE_SECRET の値>" \
  -H "Content-Type: application/json"
```

## セキュリティ上の注意点

| 項目 | 対策 |
|------|------|
| Admin 画面へのアクセス | Supabase 認証 + `app_metadata.roles: ["admin"]` で保護 |
| Admin の一時閉鎖 | Vercel Dashboard → Settings → Deployment Protection → Password Protection を ON |
| Admin ユーザーの作成 | 本番では手動作成（シードは Auth ユーザーを作らない） |
| `REVALIDATE_SECRET` | プレースホルダー（`your-secret-key-here`）のままにしない。`openssl rand -hex 32` で生成 |
| 環境変数 | GitHub リポジトリには `.env` をコミットしない（`.gitignore` 確認済み） |
| Supabase Service Role Key | Admin・Web のサーバー側のみで使用。クライアントには絶対公開しない |
| `NEXT_PUBLIC_` 変数 | ビルド時にバンドルされるため、公開して問題ない値のみ設定すること |

## まとめ・推奨アクション

1. **まず Vercel Pro + Supabase Free で試験運用開始**（月約 $21 〜 $25）
2. Vercel の Spend Management で月の上限を設定し、青天井を防ぐ

## GitHub Actions の設定

### ブランチ構成（kawasaki 版）

このリポジトリのデフォルトブランチは **`kawasaki/develop`**（元の `develop` ではない）。
GitHub Actions の各ワークフローも `kawasaki/develop` に対応させている。

| ブランチ | 役割 |
|---------|------|
| `kawasaki/develop` | 開発ブランチ（デフォルト）。PR のマージ先。staging 環境にデプロイ |
| `main` | 本番ブランチ。production 環境にデプロイ |

### ワークフロー構成

| ファイル | トリガー | 内容 |
|---------|---------|------|
| `code_check.yml` | main・kawasaki/develop へのプッシュ・PR | Biome lint・型チェック・テスト実行 |
| `deploy.yml` | main・kawasaki/develop へのプッシュ | Supabase マイグレーション適用 → Vercel デプロイ |
| `pinact_check.yml` | ワークフローファイル変更時 | Actions のバージョンがコミットSHAにピン留めされているか確認 |

### デプロイフロー（deploy.yml）

```
git push → Supabase link → supabase db push → supabase config push → Vercel Deploy Hook 呼び出し（web・admin）
```

- `main` ブランチ → `production` 環境
- `kawasaki/develop` ブランチ → `staging` 環境

### 必要な GitHub Secrets の設定

GitHub リポジトリの **Settings → Secrets and variables → Actions** に以下を登録する。

> 環境（Environment）ごとに値が異なる場合は、Secrets を「Repository secrets」ではなく **Environment secrets** として登録すること。

#### Supabase 関連

| Secret 名 | 取得場所 | 説明 |
|-----------|---------|------|
| `SUPABASE_ACCESS_TOKEN` | [Supabase Dashboard → Account → Access Tokens](https://supabase.com/dashboard/account/tokens) | Supabase CLI 認証用トークン |
| `SUPABASE_PROJECT_REF` | Supabase Dashboard → プロジェクト → Settings → General → Reference ID | プロジェクトの参照ID（例: `abcdefghijklmnop`） |
| `SUPABASE_DB_PASSWORD` | Supabase プロジェクト作成時に設定したDBパスワード（後から確認不可のため保管しておくこと） | DB 接続パスワード |

#### Vercel 関連

| Secret 名 | 取得場所 | 説明 |
|-----------|---------|------|
| `WEB_VERCEL_DEPLOY_HOOK_URL` | Vercel Dashboard → web プロジェクト → Settings → Git → Deploy Hooks | web の手動デプロイトリガー URL |
| `ADMIN_VERCEL_DEPLOY_HOOK_URL` | Vercel Dashboard → admin プロジェクト → Settings → Git → Deploy Hooks | admin の手動デプロイトリガー URL |

**Deploy Hook の作成手順:**
1. Vercel Dashboard → 対象プロジェクト → Settings → Git → Deploy Hooks
2. 「Create Hook」でフック名（例: `GitHub Actions`）とブランチを指定
3. 生成された URL を Secret に登録

### GitHub Environments の作成

`deploy.yml` は `production`（main）と `staging`（kawasaki/develop）という Environment を参照している。
リポジトリの **Settings → Environments** で以下を作成する：

| Environment 名 | 対象ブランチ | 推奨設定 |
|--------------|------------|--------|
| `production` | `main` | Required reviewers を設定し、本番デプロイに承認フローを追加することを推奨 |
| `staging` | `kawasaki/develop` | 自動デプロイで問題なし |

Environment を作成しておかないと `deploy.yml` の `environment:` 指定が機能しないため、必ず作成する。

### 注意事項

- `supabase config push` はローカルの `supabase/config.toml` の設定を本番に反映する。Auth 設定（メール送信者名など）が上書きされるため、本番側で手動変更した設定は上書きされる点に注意。
- Vercel の Deploy Hook 経由のデプロイは Vercel 側に環境変数が設定済みであることが前提。先に Vercel の環境変数を設定しておくこと。
- `NEXT_PUBLIC_*` 環境変数は**ビルド時にバンドルに組み込まれる**。変更した場合は必ず再デプロイが必要（キャッシュを使わないフルビルド）。
- `pinact_check.yml` はワークフローの Action バージョンがコミットSHAにピン留めされていることをチェックする。新しい Action を追加する際は `@v4` のような形式ではなく、`@コミットSHA # v4` の形式で記載すること。
