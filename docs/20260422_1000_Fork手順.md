# みらい議会 Lightfork — Fork 手順

本リポジトリ (`mirai-gikai-lightfork`) を別の地方議会向けに fork して立ち上げる際の手順をまとめます。

> [!NOTE]
> Lightfork は Supabase を使いません。データは `data/` 配下の JSON で管理し、Admin はローカル起動専用です。
> Web 側のユーザー発 AI 機能（チャット・インタビュー等）は全廃されています。

> [!TIP]
> **AI 初心者・Git 初心者向けの無料セットアップ版**（Windsurf + Gemini CLI）は
> [20260422_1400_Windsurf+GeminiCLIでのFork手順.md](./20260422_1400_Windsurf%2BGeminiCLIでのFork手順.md) を参照してください。
> 本ドキュメントは Claude Code を使う前提の標準版です。

## 事前準備

- Node.js 20 以上（推奨: 22）
- pnpm 10 以上
- Git
- （任意）AI CLI — Admin の AI 情報収集機能を使う場合。次のいずれか:
  - **Claude Code CLI**（本手順書の前提）
  - **Gemini CLI**（無料スタート向け。`AI_CLI_PROVIDER=gemini` を設定。詳細は [20260422_1400_Windsurf+GeminiCLIでのFork手順.md](./20260422_1400_Windsurf%2BGeminiCLIでのFork手順.md) 参照）
- （任意）Vercel アカウント — Web を公開する場合

## 1. リポジトリを Fork する

1. GitHub 上で本リポジトリを Fork、または `git clone` してから独自の新規リポジトリを作成。
2. リポジトリ名は自治体名を含めた識別しやすい名前に（例: `mirai-gikai-sample-city`）。
3. Fork 直後に必ず [FORK_GUIDELINES.md](../FORK_GUIDELINES.md) の必須要件（サービス名・ロゴ・カラー・免責文言）を確認してください。

## 2. サイト設定ファイルを書き換える

`web/src/config/site.config.ts` と `admin/src/config/site.config.ts` の 2 ファイルが対象です。両方を揃えて書き換えてください。

### Web 側（`web/src/config/site.config.ts`）

| フィールド | 説明 |
| --- | --- |
| `siteName` | ヘッダー・OGP・メタタグで使うサービス名（例: `みらい議会＠サンプル市`） |
| `siteDescription` | トップページ等のメタディスクリプション |
| `cityName` / `councilName` | 自治体名・議会名 |
| `keywords` | SEO 用キーワード配列 |
| `councilBaseUrl` | 自治体公式サイトのベース URL |
| `councilBillsDetailUrl` | 議会の議案一覧ページ URL |
| `twitterHashtag` | SNS シェア時のハッシュタグ（`#` なし） |
| `externalLinks.*` | お問い合わせ・寄付・運営元等の外部リンク |
| `managingParty` | ページを管理する政党名。未所属で運用する場合は空文字のまま。 |
| `operator.*` | サービス運営者情報（利用規約の準拠法・管轄裁判所を含む） |
| `features.showLogo` | 独自ロゴを用意してから `true` にする |
| `features.showTeamMiraiSection` | チームみらい公式でない場合は `false` のまま |

### Admin 側（`admin/src/config/site.config.ts`）

Web 側と同じ `siteName` / `cityName` / `councilName` を設定します。`councilFactionExamples` には AI 情報収集時のプロンプトに使う「会派名の例」（複数）を書いてください。

> [!IMPORTANT]
> ロジック側でサイト名・自治体名をハードコードしないこと。必ず `siteConfig` 経由で参照してください。

## 3. ロゴ・カラー・OGP 画像の差し替え

- ロゴ: `web/public/img/logo.svg` を独自ロゴに置き換え、`site.config.ts` の `features.showLogo` を `true` に。
- OGP / 画像: `web/public/` 配下の `og-image.png` 等を差し替え。
- テーマカラー: `web/src/app/globals.css` の `@theme inline` ブロックで定義済みトークンを編集。
  - インラインでの色指定（`bg-[#xxx]` / `style={{ color: '#...' }}`）は禁止。必ずトークン経由で。
- フォント・タイポグラフィ: 必要に応じて `globals.css` を編集。

## 4. サンプルデータの削除と初期データ登録

リポジトリ直下の `data/` 配下にサンプルデータが配置されています。ファイル名・議案名に `【サンプル】` を付けているので、自治体の本番運用前に削除してください。

```bash
# サンプルデータを一括削除
rm data/bills/sample-*.json
rm data/factions/sample-*.json
rm data/tags/sample-*.json
rm data/council-sessions/sample-*.json
```

### A. Admin 画面から登録する

```bash
pnpm install
pnpm dev       # web:3000, admin:3001 が並列起動
```

ブラウザで [http://localhost:3001](http://localhost:3001) を開くと、認証なしで Admin 画面に入れます。以下の順で登録してください：

1. **会期** (`council-sessions`) — 現在進行中の定例会を 1 つ作成し、`is_active = true` に
2. **会派** (`factions`) — 議会を構成する会派を登録
3. **タグ** (`tags`) — 議案の分類に使うタグ（子育て／教育／インフラ 等）
4. **議案** (`bills`) — 議案ごとに内容・公開ステータス・会派賛否を登録

### B. JSON を直接編集する

`data/{bills,factions,tags,council-sessions}/<id>.json` を直接置くこともできます。スキーマは `packages/data/src/types.ts` を参照してください。ファイル名と `id` フィールドは必ず一致させること。

> [!NOTE]
> `data/` 配下への `fs` での直接アクセスは禁止です。コードから操作する場合は必ず `@mirai-gikai/data` パッケージ経由で読み書きしてください。

### C. AI 情報収集を使う（任意）

議案の初期整備をラクにしたい場合、Admin の AI 情報収集機能で一次ドラフトを生成できます。

1. `.env` の `AI_CLI_PROVIDER` を設定
   - `gemini`（既定） — Gemini CLI をサブプロセス実行
   - `gemini-api` — Gemini API を直接呼び出し（**無料枠の RPM/RPD 上限が緩く推奨**）。[Gemini API直叩きモード](./20260423_1500_Gemini%20API%E7%9B%B4%E5%8F%A9%E3%81%8D%E3%83%A2%E3%83%BC%E3%83%89.md) を参照
   - `claude` — Claude CLI をサブプロセス実行
2. プロバイダに応じた設定
   - `gemini` : `GEMINI_CLI_PATH`（任意）
   - `gemini-api` : `GOOGLE_AI_API_KEY`（必須）
   - `claude` : `CLAUDE_CLI_PATH`（任意）
3. Admin の「AI 情報収集」画面から収集期間を指定して実行
4. 生成されたドラフトを確認・編集してから公開ステータスを `published` に切り替え

## 5. 環境変数

`.env.example` をコピーして `.env` を作成し、必要な値だけ埋めてください。

```bash
cp .env.example .env
```

最低限動かすために必要なのは次の項目だけです：

- `REVALIDATE_SECRET` — Admin → Web のキャッシュ再検証で使う共有シークレット（任意の文字列）
- `WEB_PORT` / `ADMIN_PORT` — 他の mirai-gikai 系プロジェクトと同時起動する場合

Supabase 系の環境変数は不要です。

## 6. ローカルでの開発

```bash
# 全部まとめて起動
pnpm dev

# 個別起動
pnpm --filter web dev
pnpm --filter admin dev

# テスト / 型チェック / Lint
pnpm test
pnpm typecheck
pnpm lint
```

## 7. Web の公開（デプロイ）

Web 側（`web/`）は Vercel 等へそのままデプロイできます。`data/` 配下の JSON を git 管理したまま Vercel にデプロイすれば、追加の外部 DB は不要です。

- Vercel プロジェクトを作成し、`web/` をルートディレクトリに指定
- 環境変数で `REVALIDATE_SECRET` と（必要なら）`BASIC_AUTH_USER` / `BASIC_AUTH_PASSWORD` を設定
- デプロイ後、カスタムドメインを設定

> [!IMPORTANT]
> Admin (`admin/`) は **絶対に本番デプロイしない** こと。ローカル起動専用で認証がなく、編集操作はすべて直接 `data/` に書き込まれる設計です。

## 8. 運用フロー

1. Admin をローカル起動して議案データを編集・追加
2. 変更を `git commit` / `git push`
3. Web の Vercel 側が自動デプロイされ、新しい `data/` が反映
4. （任意）Admin から `REVALIDATE_SECRET` 経由で Web の ISR キャッシュを明示的に無効化

複数人で運用する場合は、編集者ごとに自リポジトリの clone 上で Admin を起動し、PR ベースでデータ更新をレビューする運用を推奨します。

## 9. 本家 (`mirai-gikai-kawasaki`) と違う点の把握

| 項目 | 本家 | Lightfork |
| --- | --- | --- |
| データストア | Supabase | `data/` 配下の JSON |
| Admin 認証 | Google OAuth + 管理者 DB | 無認証（ローカル専用） |
| Web 側 AI 機能 | チャット・インタビュー等あり | すべて削除 |
| Admin 側 AI 情報収集 | あり | あり（維持） |

Web 側の AI 機能を追加で復活させたい場合は、本家 `mirai-gikai-kawasaki` の該当 feature を参考に移植してください。

## 10. トラブルシューティング

- **`data/` が見つからない**: `packages/data/src/paths.ts` の `getDataRoot()` は `DATA_DIR` 環境変数 → `process.cwd()/../data` の順で解決します。既定ではリポジトリ直下の `data/` が使われます。
- **議案を保存したのに Web に反映されない**: Web は unstable_cache + タグで ISR しています。`REVALIDATE_SECRET` を設定し、Admin 側の公開操作が Web の `/api/revalidate` に通知されていることを確認してください。
- **ポート競合**: `WEB_PORT` / `ADMIN_PORT` を `.env` で変更してください。
